# Kenya FNP database - project and work summary

Author: Aboubacar HEMA
Scope: Kenya Soil / Food / Nutrition / Policy (FNP) database and acquisition pipeline

This document summarises the design of the database and every piece of work carried out, with the key numbers highlighted. It is written to be a standalone record that a collaborator (or a future session) can pick up from.

---

## 1. What the project is

A configuration-driven, provenance-tracked pipeline that assembles the Kenya dataset bundle (soil, food, nutrition, policy) into a single layered DuckDB database, keyed on a master county and sub-county crosswalk and built for county-level profiling and the soil-to-nutrition analysis.

Design in one line: the crosswalk is the spine, every indicator is appended to it (never merged pairwise), heavy artefacts stay on disk as files, and the database holds the tidy indicators plus a full provenance ledger.

Key structural numbers:

| Item | Value |
| --- | --- |
| Thematic layers | **5** (geography, soil, food, health, policy) |
| Registered sources | **25** |
| Counties (master spine) | **47** |
| Sub-counties (master spine) | **290** |
| Database engine | DuckDB (single file) |
| Database schemas | core, geography, soil, food, health, policy, plus a provenance ledger |

---

## 2. Architecture

- **Master crosswalk** (`core.crosswalk_admin`): 47 counties and 290 sub-counties from COD-AB boundaries, with normalised names. Boundary layers are detected by content (feature counts and attributes), not by file name.
- **Three parallel keys**: spatial (county or sub-county), denominator (population, households, area), and provenance (publisher, mirror, licence, checksum, date).
- **Thematic schemas**: soil, food, health and policy each form their own schema. Gridded predictions, legacy polygons, measured points, survey microdata and policy text are kept distinct and never silently merged.
- **Provenance ledger** (`provenance`): one row per object with layer, source, path, checksum, byte size, status and extraction time. It is cumulative across runs.

Pipeline stages: acquire, then extract archives, then build the crosswalk, then run the transforms, then build the database.

## 6. Work carried out

### 6a. Action Plan extractor (`kenyadb/action_plan.py`)

Extracts the Kenya Food Systems and Land Use Action Plan (October 2023) from a locally-provided PDF and produces a structured workbook plus policy-layer CSVs.

What it does: records provenance (SHA-256, size, page count), extracts the full text, verifies anchor strings against that text, writes the workbook to `data/processed/action_plan/` and ten tidy CSVs to `data/external/action_plan/` (folded into the policy layer).

Key numbers:

| Item | Value |
| --- | --- |
| PDF pages | **65** |
| Verification anchors matched | **14 / 14** |
| Output workbook sheets | **9** |
| Policy CSV tables | **10** |
| 7-year budget total | **52,700** KES millions (reconciles exactly) |
| Budget line items (7-year) | **33** |
| Budget line items (2024) | **14** |
| Policy inventory entries | **38** |
| Legislative actions | **13** |
| Stakeholders | **69** |
| County nutrition points | **4** |

Two source inconsistencies were preserved and flagged rather than silently corrected:
- the 2024 budget prints a total of **1,600** KES millions while its listed line items sum to **3,750**;
- the stakeholder list comes to **69** because the source merged the Ministry of Health and the University of Nairobi under a single number (item 9); both were kept as distinct organisations.

County stunting figures named in the Plan: Kilifi **37**, West Pokot **34**, Samburu **31**, Kisumu **9**.

### 6b. Import-path fix

The new `run_all.py` imports `kenyadb.action_plan`, so `action_plan.py` must sit inside the `kenyadb` package folder (next to `crosswalk.py`), not at the project root. Placing it at the root produced an ImportError stating that the name action_plan could not be imported; moving the file into `kenyadb/` cleared it.

### 6c. Local-file handler and provenance sidecar merge

- A `local_file` handler records a user-provided file (checksum plus `ok` status) without any network call, or a `manual` gate if the file is absent.
- The `action_plan` registry entry was switched to this handler so a full run records it from the local PDF instead of trying to re-download.
- A provenance sidecar merge was added to `build_db.py`: any locally-provided source that writes a `*_provenance.json` has its prior rows replaced by one authoritative `ok` row on every build, including `--build-only`. This is what flips `action_plan` to acquired in the ledger.

Verified result: `action_plan` reads **ok** with the message "local extract: 65 pages, 10 CSV tables, anchors 14/14 verified".

### 6d. KDHS county transform (`kdhs_county` in `transforms.py`)

Survey-weighted county estimates from the KDHS 2022 recodes, ready for when the survey access is granted.

- Reads the children recode (KR) for stunting, wasting, underweight and child anaemia, and the women recode (IR) for women anaemia.
- Weights by **v005 / 1,000,000**; detects the county variable by matching its value labels to the crosswalk.
- Produces **5** indicators joined to all **47** counties.
- A real bug was fixed: metadata is read with `read_dta(metadataonly=True)`, not the non-existent `read_file_metadata`.
- Tested on synthetic **47**-county recodes (6,000 child records, 4,000 women records); the women tested-flag filter reduced the women sample to **3,611**.
- `kdhs_2022` was added to the dedicated-source set so the generic ingester leaves its microdata alone.

### 6g. Open-download audit and local PDF assessment

All open_download sources were checked, the direct download links verified, and the three locally-provided PDFs assessed for extraction.

Verified direct download links:
- KNBS 2019 census denominators publish as direct XLSX files (population, households and density by county, by sub-county, and by administrative units), which are cleaner than the PDF. Census Volume I and Volume II are also available as direct PDFs.
- Kenya Food Composition Tables 2018: the FAO PDF at `https://www.fao.org/3/I9120EN/i9120en.pdf` (**254** pages), with a Government of Kenya mirror.
- KENSOTER: the ISRIC data hub record `73e27136-9efe-49e4-af35-fd98b841d467`.
- ASTI: the data portal at `asti.cgiar.org/data` and the IFPRI Dataverse handle `1902.1/20514`; ASTI moved to FAO in 2023.
- ASPIRE and GFDx: interactive portals (official data pages recorded, export is semi-manual).

Local PDF assessment:
- **kfct_2018** (KFCT, **256** pages): cleanly machine-readable. Structure is "Table 1: Energy, proximates, minerals and vitamins" with food-group headers (a 2-digit code plus a capitalised name) and food rows (a 5-digit code, name and numeric columns). A parser prototype recovered the proximates block (**651** distinct food codes) and the minerals block of Ca, Fe, Mg, P, K, Na, Zn, Se (**648** distinct food codes). This source supports a real local extractor.
- **census_ke_ag** (2019 Agriculture Analytical Report, **156** pages): graphics and narrative, not clean tables. Most pages return no extractable text and county names appear only as rotated chart-axis labels (at most **1** county name per page). The reliable county agricultural denominators therefore come from the KNBS data tables, not this PDF.
- **kphc_2019_vol1** (Census Volume I PDF): a similar KNBS report; the clean denominators are the three KNBS XLSX files. The local PDF is kept for provenance and reference.

Conclusion adopted: for the census denominators the pipeline downloads the KNBS XLSX data tables, while the locally-provided Volume I and Agriculture PDFs are recorded for provenance and full text. The KFCT PDF is the one local source that supports a full table extractor.
